# grafana

Grafana instance for the hub cluster, deployed via the community Grafana
Operator and fronted by the OpenShift OAuth proxy. The default datasource is
the in-cluster Thanos Querier, so dashboards see the same long-term metrics
that the OpenShift monitoring stack exposes.

## Components

Resources are applied in sync-wave order so the operator and its CRDs are
ready before any custom resources are created.

| Wave | Resource | Purpose |
| ---- | -------- | ------- |
| 8  | `grafana` Namespace | Tenant for the operator and Grafana instance. |
| 9  | OperatorGroup, Subscription | Installs `grafana-operator` from `community-operators` (channel `v5`), scoped to the `grafana` namespace. |
| 10 | ServiceAccount, ClusterRoleBindings, SA token Secret, OAuth cookie Password + ExternalSecret | Identity and credentials. |
| 11 | `Grafana` CR, `GrafanaDatasource` CR | The Grafana instance and its Thanos datasource. |

## Authentication

Login uses OpenShift OAuth via a sidecar `oauth-proxy`:

- `grafana-grafana.yaml` injects a second container (`origin-oauth-proxy`)
  into the Deployment the operator manages. It terminates TLS on `:8443`
  using a service-serving cert (`service.beta.openshift.io/serving-cert-secret-name`),
  and forwards to Grafana on `127.0.0.1:3000`.
- The `grafana` ServiceAccount carries an
  `serviceaccounts.openshift.io/oauth-redirectreference.primary` annotation
  pointing at the `grafana-route`, so OpenShift treats it as an OAuth client.
- Authorization is gated by an SAR: the user must be able to `get` the
  `grafana-service` Service in the `grafana` namespace. The
  `system:auth-delegator` ClusterRoleBinding lets the proxy validate tokens
  via the TokenReview / SubjectAccessReview APIs.
- Grafana itself runs with `auth.proxy` enabled, trusting the
  `X-Forwarded-User` header set by the proxy. The local login form is
  disabled and `auto_assign_org_role: Admin` grants any authenticated
  OpenShift user admin in Grafana — access control happens at the SAR layer.
- The proxy cookie secret is a 32-character random value generated by an
  External Secrets `Password` generator and materialised into the
  `grafana-oauth-cookie` Secret.

## Route

A reencrypt Route at `grafana.apps.hub.igou.systems` targets the proxy's
`https` port. Edge traffic is HTTPS to the router, then HTTPS again from the
router to the pod using the service-serving cert.

## Datasource

`grafana-thanos-datasource.yaml` defines a Prometheus-type datasource
pointed at `https://thanos-querier.openshift-monitoring.svc:9091` and marks
it as the default. Authentication uses a bearer token sourced from the
`grafana-sa-token` Secret (a long-lived ServiceAccount token for the
`grafana` SA) injected through the operator's `valuesFrom` mechanism into
`secureJsonData.httpHeaderValue1`. The `cluster-monitoring-view`
ClusterRoleBinding gives that SA read access to all metrics exposed by
Thanos Querier.

`tlsSkipVerify: true` is set because the Thanos Querier serving cert is
signed by the internal service CA, which Grafana does not trust by default.

## Dashboards

The Grafana CR carries the label `dashboards: grafana`, and the datasource
selects instances with the same label via `instanceSelector`. Any
`GrafanaDashboard` CR created in this (or another watched) namespace with
that label will be reconciled into this instance.
